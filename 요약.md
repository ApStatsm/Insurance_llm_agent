보험 약관 AI 챗봇 기능 정리 (Phase 1~5)
========================================

작성 목적
--------

이 문서는 현재 Streamlit 기반 보험 약관 AI 챗봇 프로젝트에 구현된 기능을 Phase별로 정리한 텍스트 문서이다.
프로젝트는 고객이 보험 사고 상황이나 청구 서류를 질문하면, AI가 약관 근거와 고객 가입 정보를 바탕으로 안내하고, 업로드 서류를 점검하며, 상담원/관리자 업무 화면까지 이어지는 데모형 보험 업무 시스템을 목표로 한다.

주의 사항
--------

- 현재 시스템은 데모용이다.
- AI 답변은 보험금 지급 여부를 확정하지 않는다.
- "청구 서류 준비율"은 지급 가능성이 아니라 제출 서류 준비 상태를 뜻한다.
- 가상 접수번호는 실제 보험사 접수번호가 아니다.
- 실제 보험사 청구 API, CRM, 상담원 시스템과 연동되어 있지 않다.
- 고객 ID 입력 방식과 API 키 로딩 방식은 기존 구조를 유지한다.

전체 실행 흐름
--------------

1. 사용자가 Streamlit 화면에서 고객 ID, 질문, 첨부 파일을 입력한다.
2. `app/main.py`가 업로드 파일을 저장하고 pipeline을 호출한다.
3. `run_multi_agent_pipeline`이 다음 순서로 실행된다.
   - 고객 DB 조회
   - 질문 라우팅
   - route별 worker 실행
   - manager 검수 및 최종 답변 생성
4. worker는 기존 답변 문자열과 함께 `diagnosis_result` 구조를 생성한다.
5. 업로드 파일이 있으면 Vision 분석 또는 파일명 fallback으로 서류 점검을 수행한다.
6. 최종 응답 후 mock ticket을 생성하고 상담원 전달 요약을 저장한다.
7. 관리자 대시보드에서 저장된 ticket 현황을 확인할 수 있다.

### 목표

- 키워드 기반 라우팅 안정화
- fallback route 명확화
- 민원/상담원 키워드 우선 처리
- 라우팅 테스트 스크립트 추가
- 고객 가입상품 기반 RAG 검색 결과 재정렬
- 사용자 친화적 오류 메시지 추가


실제 외부 시스템 연동은 하지 않고 JSON/JSONL 기반으로 저장한다.

### 추가 기능

- 가상 접수번호 생성
- 상담원 확인 필요 여부 표시
- 관리자 대시보드 추가
- 상담원 확인 필요 큐 표시
- ticket 상세 보기
- 데모용 mock ticket 생성 버튼

### ticket status

```text
created                접수 생성
ai_reviewed            AI 사전진단 완료
waiting_for_documents  추가 서류 대기
needs_human_review     상담원 확인 필요
ready_for_submission   기본 청구 준비 완료
transferred_to_agent   상담원 전달 완료
closed                 종료
```

### status 계산 규칙

- `cs_complaint`이면 상담원 전달 또는 상담원 확인 필요
- 상담원 확인 필요가 True이면 `needs_human_review`
- 준비율이 없으면 `ai_reviewed`
- 준비율 80% 미만이면 `waiting_for_documents`
- 준비율 80% 이상이면 `ready_for_submission`

단, `ready_for_submission`은 지급 가능성이 아니라 기본 서류 준비 상태를 뜻한다.

### 상담원 확인 필요 판단 기준

다음 중 하나라도 해당하면 상담원 확인 필요로 판단한다.

- 민원/상담원 연결 route
- 분쟁/판례 route
- 보상 근거 부족
- 면책 가능성 있음
- 약관 근거 카드 부족
- 누락 서류 존재
- 청구 서류 준비율 80% 미만
- 고객 정보와 서류 정보 불일치 가능성
- 핵심 필드 누락
- 문서 confidence 0.6 미만
- 문서 needs_review
- 질문에 민원/분쟁/소송/금감원/상담원 키워드 포함

### priority

```text
low     낮음
medium  보통
high    높음
urgent  긴급
```

우선도 예:

- 민원/상담원 요청: `urgent`
- 분쟁/소송/금감원/면책 가능성: `high`
- 서류 누락/준비율 낮음: `medium`
- 최종 확인만 필요한 경우: `low`

### 관리자 대시보드 KPI

- 전체 접수 건수
- 오늘 접수 건수
- AI 사전진단 완료 건수
- 상담원 확인 필요 건수
- 평균 청구 서류 준비율
- 준비율 80% 이상 건수
- 민원성 문의 건수
- 문의 유형 분포
- 상품별 문의 분포
- 우선도별 상담 건수
- 자주 누락되는 서류 TOP 5

### 상담원 확인 필요 큐

필터 조건:

- `human_review_required == True`
- 또는 `status == needs_human_review`
- 또는 `priority in ["high", "urgent"]`

표시 컬럼:

- 접수번호
- 생성일시
- 고객ID
- 상품
- 문의유형
- route
- 상태
- 우선도
- 준비율
- 주요 사유

### 중복 ticket 생성 방지

Streamlit rerun으로 같은 응답이 반복 저장되지 않도록 다음 key를 사용한다.

```text
trace_id + user_id + question
```

이 key에 대해 생성된 ticket_id를 `st.session_state.ticket_keys`에 저장한다.


대표 데모 시나리오
----------------

### 1. 자동차 침수

```text
customer_id: CUST-3301
question: 태풍 때문에 차가 침수됐는데 보상 가능해?
```

예상 동작:

- route: `policy_diagnosis`
- ticket prefix: `CLM`
- 사고 유형: 차량 침수
- 상담원 확인 필요: True
- 우선도: medium 또는 high
- 주요 확인:
  - 자기차량손해 담보 확인
  - 사고사실확인서 누락
  - 차량등록증 누락
  - 침수 원인/사고 장소 확인

### 2. 실손 도수치료

```text
customer_id: CUST-1029
question: 도수치료 받았는데 실손 청구 가능해?
```

예상 동작:

- route: `policy_diagnosis` 또는 `document_claim`
- ticket prefix: `CLM` 또는 `DOC`
- 사고 유형: 실손 의료비
- 상담원 확인 필요: True
- 우선도: medium
- 주요 확인:
  - 진료비 세부내역서 필요
  - 도수치료 횟수/목적 확인
  - 비급여 항목 및 특약 확인

### 3. 암 진단비

```text
customer_id: CUST-2044
question: 암 진단비 청구하려면 뭐가 필요해?
```

예상 동작:

- route: `document_claim`
- ticket prefix: `DOC`
- 사고 유형: 암 진단비
- 상담원 확인 필요: True
- 우선도: medium 또는 high
- 주요 확인:
  - 진단서
  - 병리보고서
  - 진단 확정 요건

### 4. 민원/상담원 연결

```text
customer_id: CUST-1029
question: 상담원 연결해줘. 처리 결과가 너무 불만이야.
```

예상 동작:

- route: `cs_complaint`
- ticket prefix: `CSQ`
- 상담원 확인 필요: True
- 우선도: urgent
- 상태: 상담원 전달 완료 또는 상담원 확인 필요
- 관리자 대시보드의 민원성 문의 및 상담원 확인 필요 큐에 표시

남은 한계
---------

- 실제 보험사 시스템과 연동하지 않는다.
- 실제 접수번호가 아니라 데모용 가상 접수번호만 생성한다.
- 고객 DB에 이름/차량번호가 부족해 개인정보/차량번호 정합성 검증은 제한적이다.
- Vision 분석 품질은 실제 업로드 이미지 품질과 모델 응답에 의존한다.
- 관리자 dashboard는 기본 Streamlit 표/metric 중심이며 고급 필터나 권한 관리는 없다.
